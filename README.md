
### 🛠️ “魔改”方案：分布式演唱会抢票系统 (Ticket-Master Clone)

**不要直接用它默认的业务代码（那只是个简单的 CMS）。**
请利用它的**基础设施架构**，保留 `UserCenter`（用户中心）和 `Payment`（支付），然后**重写/新增**核心业务模块，将其包装为**“高并发演唱会抢票系统”**。

#### 为什么选“抢票”而不是“商城秒杀”？
*   **差异化**：商城秒杀（扣库存）太烂大街了。而“抢票”涉及**“选座”**、**“锁座”**、**“座位状态流转”**，业务复杂度更高，面试官更有兴趣。
*   **技术深度**：库存只是一个数字（Int），而座位是一个二维矩阵（Bitmap/Array），对数据结构和并发控制的要求更高。

### 💡 核心改造点与面试“八股”考点

你需要重点实现以下三个功能模块，每一个都对应大厂的核心考题：

#### 1. 核心功能：极致并发的“选座与锁座”
*   **业务场景**：周杰伦演唱会，体育馆 5 万个座位，100 万人同时抢。
*   **技术实现 (八股考点)**：
    *   **Redis Lua 脚本**：不要在 Go 代码里先查再改，使用 Lua 脚本保证“检查座位状态 + 锁定座位”的原子性。
    *   **Redis Bitmap / Bitfield**：如何高效存储 5 万个座位的状态？使用 Bitmap，一位代表一个座位的 0/1 状态，极大节省内存。
    *   **考点**：Redis 原子性、Lua 机制、位图数据结构。

#### 2. 核心功能：订单超时自动释放 (延迟队列)
*   **业务场景**：用户锁座后，5 分钟内未支付，系统必须自动释放座位，让其他人能买。
*   **技术实现 (八股考点)**：
    *   **方案 A (入门)**：Redis `Key Expiration` 通知（不推荐，有延迟和丢消息风险，面试可作为反例）。
    *   **方案 B (进阶)**：Redis `ZSet` 实现延迟队列。`Score` 存过期时间，轮询消费。
    *   **方案 C (高阶)**：使用 RabbitMQ 的死信队列 (DLX) 或 RocketMQ 的延迟消息。
    *   **考点**：延迟队列的 N 种实现方式对比、时间轮算法 (TimeWheel)。

#### 3. 核心功能：分布式事务 (最终一致性)
*   **业务场景**：用户支付成功（支付服务），但票务服务（Order服务）更新状态失败，导致用户付了钱没票。
*   **技术实现 (八股考点)**：
    *   **引入 DTM**：集成 `dtm-labs/dtm` 分布式事务框架。
    *   **Saga 模式**：
        1.  正向操作：扣款、出票。
        2.  补偿操作：退款、释放座位。
    *   **考点**：分布式事务理论（CAP/BASE）、2PC/3PC/TCC/Saga 的区别与选型。

### 🔥 阶段三：高并发锁座核心 (重难点 - 面试金矿) (5-7天)
**目标**：解决“超卖”问题，引入 Redis 和 Lua 脚本。

1.  **Redis 数据结构设计**：
    *   使用 **Bitmap** 或 **Set** 存储某个场次的“已占用座位”。

2.  **编写 Lua 脚本 (原子性)**：
    *   逻辑：检查座位是否被占用 -> 如果未占用，标记为占用 -> 设置过期时间（锁座 TTL）。

3.  **实现延迟队列 (自动释放)**：
    *   使用 **Redis ZSet**。
    *   Key：`ticket:order:delay_queue`，Member：订单号，Score：过期时间戳。
