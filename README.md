#### 为什么选“抢票”而不是“商城秒杀”？
*   **差异化**：商城秒杀（扣库存）太烂大街了。而“抢票”涉及**“选座”**、**“锁座”**、**“座位状态流转”**，业务复杂度更高，面试官更有兴趣。
*   **技术深度**：库存只是一个数字（Int），而座位是一个二维矩阵（Bitmap/Array），对数据结构和并发控制的要求更高。

### 💡 核心改造点与面试“八股”考点

你需要重点实现以下三个功能模块，每一个都对应大厂的核心考题：

#### 1. 核心功能：极致并发的“选座与锁座”
*   **业务场景**：周杰伦演唱会，体育馆 5 万个座位，100 万人同时抢。
*   **技术实现 (八股考点)**：
    *   **Redis Lua 脚本**：不要在 Go 代码里先查再改，使用 Lua 脚本保证“检查座位状态 + 锁定座位”的原子性。
    *   **Redis Bitmap / Bitfield**：如何高效存储 5 万个座位的状态？使用 Bitmap，一位代表一个座位的 0/1 状态，极大节省内存。
    *   **考点**：Redis 原子性、Lua 机制、位图数据结构。

#### 2. 核心功能：订单超时自动释放 (延迟队列)
*   **业务场景**：用户锁座后，5 分钟内未支付，系统必须自动释放座位，让其他人能买。
*   **技术实现 (八股考点)**：
    *   **方案 A (入门)**：Redis `Key Expiration` 通知（不推荐，有延迟和丢消息风险，面试可作为反例）。
    *   **方案 B (进阶)**：Redis `ZSet` 实现延迟队列。`Score` 存过期时间，轮询消费。
    *   **方案 C (高阶)**：使用 RabbitMQ 的死信队列 (DLX) 或 RocketMQ 的延迟消息。
    *   **考点**：延迟队列的 N 种实现方式对比、时间轮算法 (TimeWheel)。

#### 3. 核心功能：分布式事务 (最终一致性)
*   **业务场景**：用户支付成功（支付服务），但票务服务（Order服务）更新状态失败，导致用户付了钱没票。
*   **技术实现 (八股考点)**：
    *   **引入 DTM**：集成 `dtm-labs/dtm` 分布式事务框架。
    *   **Saga 模式**：
        1.  正向操作：扣款、出票。
        2.  补偿操作：退款、释放座位。
    *   **考点**：分布式事务理论（CAP/BASE）、2PC/3PC/TCC/Saga 的区别与选型。

### 🎟️ 项目名称：Go-Zero 分布式抢票系统 (Ticket-Master)
**基座项目**：[go-zero-looklook](https://github.com/Mikaelemmmm/go-zero-looklook)
**核心目标**：实现高并发下的“选座、锁座、支付、出票”全流程，保证不超卖、数据一致。

### 🗓️ 阶段一：环境搭建与基座“瘦身” (1-3天)
**目标**：跑通 `go-zero-looklook`，理解微服务基础结构，保留用户中心，剥离无关业务。

1.  **环境准备**：
    *   安装 Docker & Docker Compose。
    *   安装 Go 1.18+、Etcd、Redis、MySQL。
    *   安装 `goctl` 工具（Go-Zero 的代码生成神器）。
2.  **拉取与运行**：
    *   Clone 代码，运行 `docker-compose up -d` 启动基础设施（Nginx, Jaeger, Prometheus 等）。
    *   运行 UserCenter 服务，确保能注册、登录，获取 JWT Token。
3.  **业务清理**：
    *   保留 `usercenter`（用户服务）和 `gateway`（网关）。
    *   **新建** `ticket`（票务服务）和 `order`（订单服务）的目录结构。
    *   *（可选）删除原项目中关于 CMS、内容管理等无关代码，保持清爽。*

> **✅ 阶段一验收标准**：能通过 Postman 调用接口完成用户登录，并获得 Token。

### 🧱 阶段二：业务建模与基础 API 开发 (3-5天)
**目标**：设计数据库模型，完成基本的 CRUD，重点在于“座位”模型的设计。

1.  **数据库设计 (MySQL)**：
    *   `ticket_event`：演唱会场次表（ID, 名称, 时间）。
    *   `ticket_seat`：座位表（ID, 场次ID, 区域, 排, 号, **状态: 可选/锁定/已售**）。
    *   `order_main`：订单主表。
2.  **API 定义 (.api & .proto)**：
    *   定义 `GetSeatList` (获取座位图)、`LockSeat` (锁座)、`CreateOrder` (下单) 接口。
3.  **基础逻辑实现**：
    *   使用 `goctl` 生成代码。
    *   实现从 MySQL 查询座位列表。
    *   实现最基础的下单逻辑（先不考虑并发，直接操作数据库）。

> **✅ 阶段二验收标准**：能调用 API 查看到座位列表，能下一个普通的订单（数据库有记录）。

### 🔥 阶段三：高并发锁座核心 (重难点 - 面试金矿) (5-7天)
**目标**：解决“超卖”问题，引入 Redis 和 Lua 脚本。

1.  **Redis 数据结构设计**：
    *   使用 **Bitmap** 或 **Set** 存储某个场次的“已占用座位”。
    *   Key 示例：`ticket:event:1001:seats`。
2.  **编写 Lua 脚本 (原子性)**：
    *   逻辑：检查座位是否被占用 -> 如果未占用，标记为占用 -> 设置过期时间（锁座 TTL）。
    *   **任务**：在 Go 中通过 `redis.Eval` 调用该脚本。
3.  **实现延迟队列 (自动释放)**：
    *   使用 **Redis ZSet**。
    *   Key：`ticket:order:delay_queue`，Member：订单号，Score：过期时间戳。
    *   启动一个 Goroutine 定时轮询 ZSet，找到超时的订单 -> 恢复 Redis 座位状态 -> 修改 MySQL 订单状态为“已关闭”。

> **✅ 阶段三验收标准**：
> 1.  用 JMeter 模拟 100 个并发请求抢同一个座位，**只有一个**能成功。
> 2.  锁座后 5 分钟不支付，座位自动变回“可选”状态。

### 🛡️ 阶段四：分布式事务与资金安全 (5天)
**目标**：保证“支付成功”和“出票成功”的一致性，防止用户扣了钱没票。

1.  **集成 DTM**：
    *   部署 DTM Server (Docker)。
    *   在 Go 代码中引入 DTM SDK。
2.  **Saga 事务编排**：
    *   定义三个步骤：
        1.  **PayService.Try**：扣款（或模拟扣款）。
        2.  **TicketService.Confirm**：将座位状态从“锁定”改为“已售”。
        3.  **TicketService.Cancel** (补偿)：退款，并将座位重新释放。
3.  **异常测试**：
    *   手动模拟“扣款成功但代码 Panic”的情况，观察 DTM 是否自动触发了回滚（退款）。

> **✅ 阶段四验收标准**：整个下单链路（锁座 -> 创建订单 -> 支付 -> 出票）完全打通，且支持事务回滚。
